OpenGame turns a plain-language description into a fully playable web game. This guide walks you through installing the CLI, configuring your API key, and running your first generation — from zero to a working, browser-playable game in under five minutes.

## Before you begin

Make sure you have:

- [Node.js 20 or later](https://nodejs.org/en/download) — run `node -v` to check
- An OpenAI-compatible API key (or a local deployment of GameCoder-27B)
- A terminal and an empty folder for your game

## Installation and setup

<Steps>
  <Step title="Clone and build OpenGame">
    Install OpenGame from source while the npm package is being prepared:

    ```bash
    git clone https://github.com/leigest519/OpenGame.git
    cd OpenGame
    npm install
    npm run build
    npm link
    ```

    The `npm link` step exposes the `opengame` command globally on your `PATH`. Verify the install:

    ```bash
    opengame --version
    ```
  </Step>

  <Step title="Set your API key">
    OpenGame's agent runtime uses any OpenAI-compatible API. Export the required environment variable:

    ```bash
    export OPENAI_API_KEY="your-api-key-here"
    ```

    Two optional variables let you point to a different endpoint or model:

    ```bash
    export OPENAI_BASE_URL="https://api.openai.com/v1"   # default
    export OPENAI_MODEL="gpt-4o"                         # or swap in GameCoder-27B locally
    ```

    <Tip>
      For the best game-coding accuracy, run [GameCoder-27B](/introduction#gamecoder-27b) locally and set `OPENAI_BASE_URL` to point at it. GameCoder-27B is purpose-trained on game engine APIs, project scaffolding, and bug-fix trajectories.
    </Tip>
  </Step>

  <Step title="Configure asset providers (optional)">
    OpenGame can generate images, videos, and audio assets during game creation. Each modality is configured independently, so you only need to set up the providers you want. You can mix providers freely:

    ```bash
    export OPENGAME_IMAGE_PROVIDER=tongyi         # tongyi | doubao | openai-compat
    export OPENGAME_IMAGE_API_KEY=sk-...

    # Similarly for video, audio, and reasoning:
    # OPENGAME_VIDEO_PROVIDER, OPENGAME_VIDEO_API_KEY
    # OPENGAME_AUDIO_PROVIDER, OPENGAME_AUDIO_API_KEY
    # OPENGAME_REASONING_PROVIDER, OPENGAME_REASONING_API_KEY
    ```

    OpenGame prints a one-line provider-status banner at startup so you can confirm which modalities are active before the run begins.

    <Note>
      Asset providers are optional. OpenGame generates fully playable games without them — it just won't call out to external image or video APIs. See [API keys](/configuration/api-keys) for the full provider reference and a copy-paste `.env.example` template.
    </Note>
  </Step>

  <Step title="Create a game folder">
    OpenGame generates files into your current directory. Create an empty folder for your first game and navigate into it:

    ```bash
    mkdir snake-game && cd snake-game
    ```

    <Warning>
      Run `opengame` from inside an empty folder dedicated to your game. The agent writes and modifies files directly in the working directory.
    </Warning>
  </Step>

  <Step title="Start OpenGame and describe your game">
    Launch the interactive session:

    ```bash
    opengame
    ```

    You'll see the OpenGame welcome screen. Type your game description at the prompt:

    ```
    Build a Snake clone with WASD controls and a dark theme.
    ```

    OpenGame will:

    1. Plan the architecture — engine choice, file layout, game-loop structure, and state management.
    2. Scaffold the project using **Template Skill** (selects the right engine and project skeleton).
    3. Write the code across all necessary files.
    4. Run the game in a sandboxed browser and use **Debug Skill** to catch and fix integration errors automatically.
    5. Stop when the game is playable end-to-end.

    When done, open the printed local URL or the generated `index.html` in your browser to play.
  </Step>
</Steps>

## Iterate on your game

Once the first generation is complete, keep the session open and describe what you want to change. OpenGame edits the relevant files, re-runs the game, and verifies the change:

```
add a power-up that doubles the snake's speed for 5 seconds
```

```
make the background a starfield instead of solid black
```

```
the snake passes through itself when going fast — fix that
```

Each message triggers a targeted edit-and-verify cycle rather than a full regeneration, so iterations are fast.

## Headless mode

For scripts, CI pipelines, and batch generation, skip the interactive UI and pass the prompt directly:

```bash
opengame -p "Build a memory-matching card game with 16 cards and a 60-second timer."
```

In headless mode, OpenGame automatically elevates the approval mode to `auto-edit` so it can write and modify files without waiting for confirmation. To also allow shell commands (installing dependencies, running the dev server), add `--yolo`:

```bash
opengame -p "Build a Snake clone with WASD controls and a dark theme." --yolo
```

If you want to generate games in a directory outside the OpenGame repo, set absolute paths for the template and docs stores:

```bash
export GAME_TEMPLATES_DIR="/absolute/path/to/OpenGame/agent-test/templates"
export GAME_DOCS_DIR="/absolute/path/to/OpenGame/agent-test/docs"
```

See [Headless mode](/features/headless) for the full reference.

## Essential commands

Once inside an interactive session, use these commands to control OpenGame:

| Command       | What it does                                        |
| ------------- | --------------------------------------------------- |
| `/help`       | Show available commands and keyboard shortcuts      |
| `/auth`       | Switch authentication method or update your API key |
| `/compress`   | Replace chat history with a summary to save tokens  |
| `/clear`      | Clear the terminal screen                           |
| `/theme`      | Change the OpenGame visual theme                    |
| `/language`   | View or change language settings                    |
| `/quit`       | Exit OpenGame                                       |

See the [commands reference](/features/commands) for the full list.

## Pro tips for game prompts

Prompt quality has a direct impact on output quality. These patterns consistently produce better results:

<AccordionGroup>
  <Accordion title="Be specific about gameplay mechanics">
    Vague prompts produce vague games. Include controls, win/loss conditions, enemy behavior, and any special mechanics.

    Instead of:
    ```
    make a platformer
    ```

    Try:
    ```
    Build a single-screen platformer with arrow-key controls, double jump, 3 enemies that
    patrol left-right, and a goal flag at the top-right. Player dies on contact with enemies.
    ```
  </Accordion>

  <Accordion title="Specify the engine if you have a preference">
    OpenGame picks the engine automatically, but you can override the choice:

    ```
    Use Phaser 3 for this.
    ```

    ```
    Plain HTML5 canvas, no external libraries.
    ```

    ```
    Use three.js for a 3D first-person view.
    ```

    Supported engines are HTML5 canvas, Phaser, and three.js. See [Game Skill](/game-skill) for how the selection works.
  </Accordion>

  <Accordion title="Describe the art style">
    OpenGame can target specific visual aesthetics if you describe them:

    ```
    90s Capcom arcade pixel art — high-contrast colors, no anti-aliasing.
    ```

    ```
    Pastel kawaii hand-drawn style with rounded shapes.
    ```

    ```
    Metallic sci-fi pixel art with a dark background.
    ```
  </Accordion>

  <Accordion title="Iterate one feature at a time">
    Make the core game loop work first, then add polish. Trying to specify sound, particles, menus, multiple levels, and a high-score board in the first prompt increases the chance of integration errors.

    Start with:
    ```
    Build a Snake clone with WASD controls and a dark theme.
    ```

    Then iterate:
    ```
    add a high-score display in the top-right corner
    add a start screen with the game title and a "Press Enter to play" prompt
    add a crunch sound effect when the snake eats food
    ```
  </Accordion>
</AccordionGroup>
